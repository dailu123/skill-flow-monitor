# 用 AI 找硬编码 —— 一页纸（给管理层）

## 一句话

> **不是"有没有用 AI"的区别——是 AI 凭感觉乱翻、还是 AI 按一份专家手册系统地翻。**
> 裸 Copilot 演示好看,但真上几千万行会漏、会错、每次答案还不一样;装上我们这个 skill,
> **同一个 Copilot** 就能扛真实规模、结果可复现可审计、还能一键复制给每个团队。

*(背景:在百万级 AS/400 老代码里,把"业务码"写死进逻辑的地方,在转 Java 重写时必须一处不漏地核对,
否则就是生产事故。靠人逐行看不可能;靠裸 AI 看不可靠。)*

---

## 没 skill 也能跑,有 skill 强在哪(看三个词:找得全 / 找得准 / 每次一样)

| | 裸 Copilot(没 skill) | 装了我们的 skill |
|---|---|---|
| **找得全(召回)** | 上下文有限,几千万行读不完,只能看几个文件 → 大面积漏 | 让 AI **先生成一个搜索脚本把候选全捞出来**,扛千万行 |
| **找得准(精度)** | 不懂业务规则:把注释、字段间赋值、字符串拼接当命中;又漏掉十六进制写法、命名常量 | 把专家规则写进 skill,AI **逐个判断 + 分 高/中/低 置信度** |
| **每次一样(可复现/可审计)** | 同一问题每次答案不同,没法做重写验收 | 捞候选确定性、判断有据,**结果稳定可审计** |
| **能复制** | 各人问法不同、质量参差 | 一个文件夹丢进仓库,**全团队同一套质量** |

**类比**:没 skill = 让聪明新人"去把所有硬编码找出来",不给任何说明——能找到一些,但漏一大半,
每次做法还不一样。有 skill = 给他**一份精确的检查清单 + 一把电动工具**——又快、又全、又稳,换谁做都一致。

---

## 怎么做到的(skill 三步,都写在一个 Markdown 里,人人能看能改)

1. **捞全候选**:skill 用**精确的自然语言**规定细节,让 AI **自己生成并运行一个搜索脚本**
   (默认 Windows 自带 PowerShell、零安装;老式 EBCDIC 文件就改用 Python),把可疑行全导出成一张表。
2. **逐个判断**:AI 对每一处判"是不是硬编码 + 理由 + 置信度"(比较/赋值/常量算;注释、字段间、拼接不算)。
3. **出清单**:确认命中的完整表格 + 汇总。

> skill 里**不写死代码**,只写"要做什么";AI 按规格生成脚本。**任何规则,人都能直接改这份文字。**

---

## 现场演示(3 分钟,最有说服力)

在**同一个小样本文件夹**上当场对比:
1. **裸跑**:让 Copilot"找出这文件夹里的硬编码"→ 漏几处、混进噪声、再问一次答案还不一样(故意问两次)。
2. **装 skill 跑**:`/find-hardcodes` → 自动捞出 N 个候选 → 逐个判断 → 出完整表(带置信度)。
3. **对标准答案打分**:这文件夹真实有 X 处,裸跑找到 a 处、skill 找到 b 处(b≈X)。一张对比表收尾。

---

## 我需要准备的数据(就这 5 样)

1. **演示样本文件夹**:~20–50 个**真实源文件**,**故意覆盖 5 种花样**——普通比较、十六进制写法、
   命名常量、注释里出现值、陷阱(字段间赋值 / 拼接)。这样才能展示"该找的找到、该排的排掉"。
2. **标准答案**:这份样本里**到底有几处真硬编码**(用确定性工具跑一遍 + 人工核)。**打分对比的依据,最关键。**
3. **对照组截图**:裸 Copilot 在同一文件夹的结果(把漏的/错的圈出来)。
4. **配置 & 业务码清单**:字段名规则、主机字符集、以及那批"要管的业务码"(来自数据库 DISTINCT)。
5. **环境就绪**:装好 skill 的仓库、Copilot Agent 模式可用。(怕现场翻车就用"直接粘贴"零配置版。)

---

## 怎么推广(零门槛)

> 把一个文件夹放进仓库的 `.github/skills/`,在 Copilot Chat 打 `/find-hardcodes` 就能用;
> 改最上面的配置块选扫哪些、找什么、加排除。**默认只用 Windows 自带 PowerShell,不装任何东西。**

## 诚实边界(主动讲,反而加分)

结果是**给人复核的候选,不保证 100%**——但它把"靠运气"变成"靠流程":可复现、可审计、可复制,
这正是能上生产、能验收的关键。捞候选是确定性的;最终判断是 AI 的尽力而为,留人把关。
