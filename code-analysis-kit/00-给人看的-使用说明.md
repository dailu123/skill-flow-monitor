# 双库对比分析套件（MCA-Java vs HUB-AS400，百万行级）

目标：用上下文有限的模型（Copilot/GPT/Gemini）自主分析两个百万行库，
各自汇总 → 交叉对比 → 业务报告，全程在 skillflow-monitor 上实时可视化，
且每一步留下可印证的中间产物。

## 产物一览
    code-analysis-kit/
      work/mca/  work/hub/      地图 MAP.md + 任务队列 PROGRESS.md（断点，模型的记忆指针）
      notes/mca/ notes/hub/     每单元一份理解笔记（带 文件:行号 证据）
      evidence/                 中间产物：接口/表清单 CSV、业务功能清单、能力对齐矩阵 ← 印证报告用
      reports/                  ARCH-mca.md / ARCH-hub.md / COMPARE.md / BUSINESS.md

## 第 0 步：放置
本套件随 skill-flow-monitor 仓库分发（位于仓库内 `code-analysis-kit/`）。
在 Windows 上 clone 本仓库后，下面的命令都在**仓库根目录**执行；
被分析的 mca / hub 代码库放在任意路径，用 --root 指过去即可。
把 `dot-github/copilot-instructions.md` 复制为你打开 Copilot 的那个工作区的
`.github\copilot-instructions.md`（Windows 资源管理器建不了 `.github`，用 `mkdir .github`）。

## 第 1 步：两个仓库各 bootstrap 一次（人执行）
    python code-analysis-kit\bootstrap.py --name mca --root D:\code\mca --preset java
    python code-analysis-kit\bootstrap.py --name hub --root D:\code\hub --preset as400
说明：
- as400 预设覆盖 rpgle/clp/cbl/dds/pf/lf/dspf 等；扩展名不符就用 --exts 自定义。
- 100% 文件进队列：小目录自动归并进父级单元（--min 只是"独立单元"门槛），不丢任何文件。
- 同时产出 work\<name>\SYMBOLS.txt 符号索引（类/REST端点/表/RPG调用链/DDS记录格式），
  地图按真实代码行数排序，模型可 grep 索引带着问题读源码。
- AS/400 源码常是一个目录几千个 member：超过 --chunk（默认50）个文件的目录会自动
  拆块，文件清单写在 work\hub\chunks\Uxxx.txt，模型一次只啃一块。

## 第 1.5 步：千万行级仓库先分诊（强烈推荐，尤其 AS/400 侧）
    python code-analysis-kit\triage.py --name hub
    python code-analysis-kit\triage.py --name mca
不重置进度（DONE 行不碰），只改写 TODO 行：PF/DDS 表定义→机器直接提取成
evidence\<name>\tables.csv（LLM 一行都不用读）；克隆程序→每族只读一个代表；
零引用疑似死代码→降级为轻量登记。削减统计见 work\<name>\TRIAGE.md。
随时可重跑（幂等），分析中途跑也安全。

## 第 2 步：起可视化（人执行，开两个终端，全程放着不管）
    :: 终端A：起页面（仓库根目录）
    npm install && npm run dev
    :: 终端B：每 2 秒把进度刷成 JSON
    python code-analysis-kit\render_status.py --out-dir public\status --watch 2
浏览器开 http://localhost:5173 ：
- **Skill Pipeline 页** = 双分支总览（MCA分支、HUB分支 → 交叉对比 → 业务报告），随进度变色。
  想看某一边的目录级地图：`--view mca` 或 `--view hub`。
- **ETL Lineage 页** = 缺口图：默认只画 parity≠both 的能力（仅一边有/未确认的才值得用图盯；
  全量映射看矩阵页）。要全画：render_status 加 --lineage all。
- **能力对齐矩阵** = http://localhost:5173/status/matrix.html ——映射关系的主视图：
  全量能力 × 两边模块/证据，按域分组、按对齐状态着色、可搜索过滤。capabilities.csv 一变就重新生成，F5 即新。

## 第 3 步：让模型开跑（启动咒语）
两边可以开两个会话分头跑。发给模型：

    读取 code-analysis-kit/dot-github/copilot-instructions.md 和 code-analysis-kit/work/mca/PROGRESS.md，
    按 LOOP PROTOCOL 自主执行：每完成一个单元就更新 PROGRESS、追加 evidence CSV、
    刷新可视化，然后继续下一个，直到队列清空，再按宪法进入汇总/对比/业务报告阶段。
    不要问我问题，歧义记入笔记的“假设与存疑”。

（hub 会话把路径里的 mca 换成 hub。）

## 第 4 步（推荐）：无人值守驱动器——解决"Copilot 跑几个就停"
Agent 模式有单条消息的工具调用上限，跑几个单元就会停下来等你点继续。治本方法是
外部循环驱动：每轮发起"干 5 个单元就退出"的小任务，干完再发下一轮，停不停由脚本决定。

    :: 执行器二选一（一次性安装）：
    ::   Copilot CLI：npm install -g @github/copilot   （用你已有的 Copilot 订阅，跑 copilot 登录）
    ::   Gemini CLI ：npm install -g @google/gemini-cli（免费，Google 账号登录，自带 Flash）
    python code-analysis-kit\drive.py --repo mca
    python code-analysis-kit\drive.py --repo hub --profile gemini   :: 另一个终端，并行
注意：VS Code 里的 Copilot Chat 面板无法被脚本驱动，驱动器必须走 CLI。

驱动器行为：每轮全新上下文（质量稳定不衰减）；崩掉的 IN_PROGRESS 自动重置回 TODO；
连续 2 轮没进展自动报警退出；日志在 work\<repo>\driver.log。挂机过夜的正确姿势。

仍想在 VS Code 聊天窗口里跑的话，先调两个设置缓解：
    "chat.agent.maxRequests": 999
    "chat.tools.autoApprove": true

## 怎么相信结果（印证，人只需偶尔做）
1. 看 `evidence/README.md` 的抽查方法：随机抽 inventory.csv 几行，开文件对行号。
2. 终端B 若打印 `⚠未在队列中`，说明模型在 capabilities.csv 里编了不存在的模块——让它重做该行。
3. 报告里的所有统计数字都应能用 CSV 行数复算（README 里有一行命令）。
4. Lineage 页的红色节点（单边缺失）逐个人工复核——这是对比报告的核心结论。

## 防跑飞
- 一轮一个单元；AS/400 大目录已自动拆块。
- 模型断了/换会话：重发启动咒语即可，它读 PROGRESS.md 自动续跑。
- 优先用能自己读写文件跑命令的工具（Copilot Agent 模式/Cline/Cursor）才是真自主；
  网页版 GPT/Gemini 只能当“换脑子求助”的副手。
